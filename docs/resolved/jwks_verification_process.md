# JWT 서명 검증과 JWKS (JSON Web Key Set)의 동작 원리

본 문서는 `log-doctor-provider-back` (백엔드)이 프론트엔드로부터 전달받은 파편화된 구조(OBO 및 MFA 테넌트 이슈 등) 속에서도, 어떻게 로그인 토큰(JWT)의 위조 여부를 검증하고 인증(Authentication)을 수행하는지 그 핵심 메커니즘인 'JWKS'를 정리한 문서입니다.

---

## 🔐 1. JWKS (JSON Web Key Set)란 무엇인가?

프론트엔드(Teams 앱)가 백엔드로 보내는 로그인 정보 조각인 JWT(JSON Web Token)는 누구나 내용을 까볼 수 있습니다. 하지만 이 내용이 중간에 해커에 의해 변조되지 않았다는 것을 증명하기 위해, 가장 뒤에 **'전자 서명(Signature)'**이 붙어 있습니다.

이 서명은 마이크로소프트의 서버가 **자신들만 몰래 가지고 있는 비밀키(Private Key)**로 찍어낸 아주 특별한 암호입니다.

백엔드 서버가 *"어? 이 토큰 진짜 마이크로소프트가 찍어준 게 맞나?"* 확인하려면, 비밀키와 짝을 이루는 공개 열쇠 꾸러미가 필요한데, 이 꾸러미의 이름이 바로 **JWKS (JSON Web Key Set)** 입니다.

---

## ⚙️ 2. 백엔드(FastAPI)의 JWKS 자동 검증 프로세스

현재 백엔드 프로젝트에서는 개발자가 직접 통신을 짜서 열쇠를 받아오지 않습니다. 
`app/core/security.py` 파일에 설정된 **`fastapi-azure-auth`** 라이브러리가 뒤에서 완벽하게 자동화된 5단계 프로세스를 수행합니다.

### 1단계: 테넌트 주소 추적
- 서버가 켜질 때, `.env`에 설정된 `TENANT_ID` (예: `ccdcba04...`)를 읽어들입니다.
- 이 ID를 가지고 마이크로소프트의 오픈아이디 설정 주소로 요청을 보냅니다.
  `(예: https://login.microsoftonline.com/{TENANT_ID}/v2.0/.well-known/openid-configuration)`

### 2단계: JWKS(공개키 꾸러미) 다운로드 및 캐싱
- 마이크로소프트 서버가 *"우리 동네(해당 테넌트)에서 발급한 진짜 토큰을 검증하려면 이 주소에서 열쇠를 받아가!"* 라고 JWKS 주소(`.../discovery/v2.0/keys`)를 알려줍니다.
- 백엔드(fastapi-azure-auth)는 그 주소에 접속하여 수많은 **실물 공개 열쇠(Public Keys) 뭉치를 한꺼번에 다운로드** 받습니다.
- 매번 마이크로소프트 통신비용(지연 시간)을 아끼기 위해, 백엔드 서버는 **이 열쇠 꾸러미를 본인의 메모리(캐시)에 저장해 둡니다.**

### 3단계: 프론트엔드 JWT 토큰 도착
- 사용자가 Teams(웹)에서 로그인에 성공하고 백엔드 API를 호출할 때, `Authorization: Bearer eyJhb...` 형태의 긴 토큰을 보냅니다.
- 백엔드는 이 긴 토큰 문자열을 쪼개서 `Header`, `Payload`, `Signature` 3덩이로 분리합니다.

### 4단계: 퍼즐 맞추기 (서명 검증)
- 토큰의 `Header` 부분에는 여러 개의 JWKS 열쇠 중에 정확히 **어떤 열쇠(`kid`: Key ID)로 서명했는지 아이디표가 붙어 있습니다.**
- 백엔드는 메모리(캐시)에 쌓아둔 수많은 마이크로소프트의 열쇠 꾸러미 중에서 해당 `kid`와 정확히 일치하는 자물쇠 열쇠(Public Key) 하나를 꺼냅니다.

### 5단계: 최종 자격 증명 디코딩
- 뽑아낸 마이크로소프트의 진짜 열쇠로 토큰의 전자 서명(Signature)을 풀어봅니다.
- 만약 누군가 중간에 토큰 내용을 아주 조금이라도(단 한 글자라도) 위조했다면, **열쇠가 맞물리지 않아 (서명 불일치) 백엔드는 즉시 `401 Unauthorized` 에러를 뱉고 쫓아냅니다.**
- 또한, 만료 기한(`exp`)이 지났는지, 대상(`aud`: Audience)이 우리 서버(`CLIENT_ID`)가 맞는 지도 이 도구에서 100% 동시에 검출합니다.

## 🎯 3. 요약 및 의의

저희가 맞닥뜨렸던 각종 Tenant 에러나 MFA 권한 에러들은 모두 **마지막 5단계의 JWKS 검증을 100% 무사히 완벽하게 통과한 직후**에 발생한 것입니다.
즉, 프론트엔드가 건네준 JWT 토큰 자체는 한 점의 위조나 변조도 없는 완벽하게 깨끗한 마이크로소프트의 보증 수표였다는 뜻입니다.

1. **JWKS Fetching:** 마이크로소프트의 열쇠 꾸러미를 서버가 메모리에 다운받아 들고 있음.
2. **Offline Validation:** 토큰이 들어올 때마다 매번 마이크로소프트 서버에 물어보지 않고, **로컬 서버 메모리에 있는 가져다 놓은 열쇠(JWKS)만으로 0.001초 만에 위조 여부를 색출함.**
3. **fastapi-azure-auth:** 이 복잡한 암호학적 과정을 코드 단 두 줄로 구현해 주고 있는 매우 훌륭한 라이브러리.
